Website Outline

The implemented website is titled Terrain UK. Terrain UK provides users with a tool for easily visualizing and navigating depth information across the UK. Furthermore, this tool enables users to select regions of the UK, with resizing and resolution options, and download the image for their own use. This could vary from simple planning of a hiking trip against different terrains to research and development reasons, perhaps generating 3D depth models from the provided data. In order to deliver this content, image manipulation of around 3000 ascii files for the whole UK mainland freely provided by Ordinance Survey was required extensively. As such, the development of this website focussed most heavily on the server-side. The following sections will outline the implemented functionality as well as the process of development for both the server and client.

 

Server

 

The largest amount of work on the backend has been the implementation of the mapMaker module. This is the module that is able to take in coordinates of areas of the UK and output an array representing the region's terrain using free data (~3000 ascii files for the whole UK mainland) provided by Ordnance Survey. The height data is rescaled so that the highest point in the UK is set to white on a PNG image and the lowest point is set to black. This module is also able to resize the output array to any size, larger or smaller (though clearly creation of a larger image results in estimation of data points) meaning that output PNG images can be created for any desired size.

This module also contains around 7000 200x200 pngs used to create the navigable depth map seen on the website. These images were created by a python script which would repeatedly make calls to our node server API and create a well-structured folder layout in which the images are stored.

The other module implemented in the modules folder is the conv module. This is not our code but is MIT licensed and available for free online. This module provides calculations for geocoding, specifically for this project we use it to convert between the coordinate system used by mapping APIs (latitude and longitude) to the coordinate system used by the OS map data (Easting and Northing). We included this code to avoid having to make use of commercial geocoding APIs like those provided by Google and OS maps since these need to be paid for at a certain number of requests and we do not need the majority of the large range of services they provide.

Both of the above modules are tightly linked to the backend API calls (all calls return as JSON objects unless otherwise specified):

 

/map - Pass (OS) coordinates of a square region of the uk as well as optionally the size of the output image you would like. The call returns an array which is then constructed into a PNG by the browser using the HTML5 canvas. Only returning an array and not creating a PNG to send saves huge amounts of computation time on the server.

 

/mappng - Used to return pre generated 256x256 PNG tiles at varying zoom levels so that mapping applications can create an easy to navigate view of the whole UK mainland at varying zoom levels. The PNGs were generated by using the /map API call repeatedly for large chunks of the UK and combining them in photoshop to create a 13200x26000 size PNG image. This image was then broken down into 256x256 chunks for varying levels of zoom and put into a structured folder layout.

 

/latLonToOsGrid - Used to convert a given latitude and longitude point to the corresponding point in easting and northing coordinates.

 

/osGridToLatLon - Used to convert a given easting and northing point to the corresponding point in latitude and longitude.

 

/osGridToLatLonSquare - Takes in square coordinates in northing and easting format and generates the latitude and longitude coordinates of each of the corners of the square. This call is useful for the frontend as it means only making one API call rather than four.

 

The rest of the server is a stack using Node.js, Express and Angular. We chose to use express because it added simple routing and support for various easily added middleware. Express massively reduced the amount of code we needed to write which let us focus on building a better product. It also contained graceful error handling which allowed our server not to crash upon experiencing errors.

 

Client

There are three navigable pages on this website. The first page is responsible for introducing the website and catching the users attention. The second page enables navigation of an interactive map along with an adjustable, movable polygon shape for extracting and saving depth images within the shapes region. The third page enables navigation of the UK in depth map form re-constructed from Ordinance Survey data.

As this website is attempting to provide a modern feature-rich presentation, focus has been placed on the use of features well-supported across modern browsers. As this tool is not constrained by commercial requirements to maximise market exposure, compatibility for Internet Explorer 8 and down has not been explored. Browsers used during development and testing include Firefox, Chrome, Microsoft Edge and Internet Explorer 9-11.

All html code was run through the W3C validator as well as the Total Validator program, testing for Polyglot html5. All html code was successfully validated.

 

Design

In order to develop an effective design, an initial rendering of the first and front page was produced in Gimp. Given very limited experience with regards to photo manipulation, a fair bit of practise was required to become comfortable with the Gimp environment as well as the general approach to image manipulation. The Gimp file for the front page is included in this submission and is called homePageDesign.xcf. The process for designing the front page was as follows:

1.    Set up a Gimp document using the 960 grid structure (provides grid guidelines for bootstraps responsive grid design).

2.    Generated individual layers for each section of the front page (Title, Introduction and Locations). These names for sections are consistent in the html structure for the front page.

3.    The title section was created first, consisting mostly of simple png image manipulation. Additional features such as the button were handled by generating a new layer using the rectangle select tool and filling the layer. A snapshot of the map component of the map page was taken and resized to be a central feature of the front page. To give the illusion that the snapshot was in front of the background, a filter was applied using Gimps drop shadow filter with a radius of 15. Manipulation of the shadow layer produced an effective illusion of depth.

4.    The introduction section evidently has our logo icon. This was originally designed to be static and was designed in Gimp through use of the free select tool and stroke selection. Following selection of a polygon, it was filled to the appropriate colour to give the effect of depth. Text was simply added using the text tool. All other images found in the design were taken from Google images.

Once the design was completed, assets were individually extracted. Extracting layers proved difficult initially due to limited means of exporting individual layers. However, research indicated copying and using the create from clipboard would enable extraction of layers against a locally sized background. As such, extracted layers were correctly sized for direct porting to the website.

The two map pages were not designed in Gimp due to map object constituting their core design.

Across the websites design, responsiveness for multiple device sizes has been implemented, ensuring CSS styling maintains the dynamic styling of the Bootstrap framework.

The following sections briefly outline the tasks and key features of each of the pages.

 

Home Page

The home page constitutes the first interaction with the user. As such, the content is kept brief, using visual animations and effective design to guide the user. In order to create a sense of depth, a parallax effect has been achieved by fixing the background image and forcing its z index to be less than all other content. As such, navigation through the page gives the sense of the page moving over the background. Placement of the snapshot with the shadow effect further creates this sense of depth.

We make use of SVG elements on our homepage in our logo. This logo was initially designed as a png image in GIMP but we decided to recreate it as an SVG image to allow scaling to any resolution as well as logo animation. Inkscape was initially considered for generating the SVG, however the simplicity of the polygonal design meant direct coordinates could be specified along with fill values to the SVG. Note that when the mouse cursor is hovered on the logo many elements move independently. We tried to implement this in various ways and ran into a number of problems. Initially we planned to animate SVG images using the built in SMIL animation, but this proved problematic for linking with events and browsers raised depreciated warnings. We eventually settled on using CSS3 animations which, when implemented properly, worked on all browsers but for Microsoft browsers. This was due to the fact that edge and IE will not perform animations on SVG elements. To fix this we had to split the single SVG image into a group of divs with each independently moving element as its own image.

 

Normal Map Page

The normal map page makes use of Googles map API. This was chosen after a fair amount of research regarding map tools, with initial implementation experimenting with the open-source tool LeafletJS. This research was required due to plans to generate custom tiles from depth data. Googles map API however provides a means of doing this and provide strong documentation regarding the use of polygonal selection regions, a feature very important for this page.

The "Normal Map" page lets a user drag and resize a square around a map to select an area they would like to export as a heightmap. Upon setting parameters and clicking export, the server is triggered to generate an array from the supplied bounding box coordinates. The call however only returns an array and therefore the page must generate the PNG from this array if the user is to be given a nice looking image. We avoided generating PNGs server side as they can take a very long time and would hog computation time on the server. The page calls the /map api and though clever use of the HTML canvas it is able to create a PNG image using just javascript executed in the browser, saving huge amounts of computation time on the server. Conversion of these coordinates to the depth data coordinate system makes extensive use of the coordinate conversion API. As referenced earlier, the server makes use of the HTML5 canvas, drawing the returned array to an HTML5 canvas, which is displayed via a modal display using Bootstraps framework, triggered via a JavaScript call. The coordinates of the square are primarily stored in Northing and Easting format to ensure that a heightmap can be more easily generated (since the resulting call to the mapMaker module is a square).

Prior to returning the depth image, an animated SVG is displayed in a modal briefly using CSS3 animations to indicate to the user that the server is processing.

Note that the red square on the page, even though it is defined square in Northing Easting coordinates, is skewed and changes shape as it is dragged around. This is because to be presented on the map it must be converted to latitude and longitude coordinates and the difference in coordinate systems is displayed in the shape of the square.

The large number of API calls made by this page also means that a large amount of the code ends up being quite complex due to the need to deal with asynchronicity. This was achieved by using the angular construct of promises which took some time to learn about.

 

Depth Map Page

The "Depth Map" page allows you to navigate the full resolution imagery of the UK that we have created using the depth map ascii data provided by OS Maps. The map was implemented using Google Maps API with our own custom tiling, generating tiles of varying size for each zoom level to enable maximum interactivity. We were unfortunately unable to integrate our map with the Google Maps road map due to the difference in coordinate systems. A possible future extension could be the full conversion of the depth data into Googles coordinate system.
